# SESSION管理指南

## 核心理念

SESSION管理采用**灵活建议**而非**强制规定**。没有"唯一正确"的方式。

## SESSION_ID的作用

- **复用SESSION**：模型记住之前的对话上下文
- **新开SESSION**：模型从零开始，无历史包袱

## 决策框架

### 快速判断

在调用Codex/Gemini前问自己：

> "如果模型'记得'之前的讨论，会**帮助**还是**干扰**？"

| 答案 | 建议 |
|------|------|
| 帮助 | 复用SESSION |
| 干扰 | 新开SESSION |
| 不确定 | 默认新开（更安全） |

### 场景示例

#### 推荐新开SESSION

| 场景 | 理由 |
|------|------|
| Code Review | 需要独立客观视角，避免先入为主 |
| 探索不同方案 | 避免前一方案的假设影响新方案 |
| 上下文过长 | 模型可能"记混"早期内容 |
| 并行worktree | 各worktree任务独立 |
| 职责切换 | 后端→前端，上下文不相关 |

#### 推荐复用SESSION

| 场景 | 理由 |
|------|------|
| 迭代修复 | 基于上轮反馈继续优化 |
| 深入探讨 | 累积理解深度 |
| 上下文依赖 | 需要理解之前的决策背景 |

## 同一Story多会话

一个Story完全可以有多个独立会话：

```
Story: 实现用户登录
├── 会话1: 主线实现（复用）
├── 会话2: 备选方案探索（新开）
├── 会话3: Code Review（新开，独立视角）
└── 会话4: 修复Review问题（可复用会话1或新开）
```

## 不强制格式

SESSION_ID没有强制格式要求：
- ✅ `abc123`（系统生成）
- ✅ `story-1-impl`（自定义）
- ✅ `auth-feature`（自定义）

工具返回的SESSION_ID可以直接使用，也可以忽略新开。

## 与BMAD工作流集成

| BMAD阶段 | SESSION建议 |
|----------|-------------|
| create-architecture | 按讨论主题复用 |
| dev-story | 按Story复用，Review新开 |
| code-review | 强制新开，独立视角 |
| correct-course | 继承当前Story会话 |
| retrospective | 新开，全局视角 |

## 实践建议

1. **不要过度管理**：简单任务不需要刻意规划SESSION
2. **信任直觉**：如果感觉需要"清空"上下文，就新开
3. **记录重要会话**：对关键决策的SESSION_ID可以记录备用
4. **灵活调整**：没有完美策略，根据实际情况调整
